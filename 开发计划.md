# MyWeb 开发计划文档（重构重开版）

**版本**: v2.0  
**日期**: 2026-03-24  
**依据**: `需求文档.md`、`tech-spec.md`、`docs/开发文档/`*  
**目标**: 从框架重整开始，按模块闭环开发，确保每一步可测试、可提交、可复现。

---

## 1. 总体执行原则（强制）

1. 一次只推进一个主模块，不跨模块并行开发。
2. 每个模块必须完成完整闭环：`任务拆分 -> 实现 -> 自测 -> 联调 -> 回归 -> 文档更新 -> Git 提交`。
3. 测试未通过，不允许切入下一模块。
4. 禁止硬编码密钥，所有敏感配置仅允许环境变量注入。
5. 每次提交都必须保证其他成员可在新环境复现（启动、测试、关键流程可跑通）。

---

## 2. 统一完成定义（DoD）

每个模块完成时必须同时满足以下条件：

- 功能完成：模块范围内所有功能点全部实现，且无已知阻断缺陷。
- 测试完成：单元/集成测试通过，关键路径有可重复执行的验证结果。
- 联调完成：前后端核心流程走通（列表、详情、创建、更新、删除、查询）。
- 回归完成：已完成功能无明显回退（至少执行一次冒烟回归）。
- 文档完成：`开发计划.md` 和对应模块文档、日程文档同步更新。
- 提交完成：按 Conventional Commits 提交到本地仓库，提交信息明确变更目的。

---

## 3. 统一测试与提交流程（每模块必做）

### 3.1 测试流程

1. 后端测试：执行 `mvn test`，要求全量通过。
2. 前端构建：执行 `npm run build`，要求构建通过。
3. 核心手测：对本模块关键用户路径执行至少 1 轮人工验证。
4. 边界验证：至少覆盖成功、失败、边界参数三类输入。
5. 回归验证：验证首页、导航、健康检查、已完成核心模块不受影响。

### 3.2 Git 提交流程

1. 仅暂存本模块相关文件。
2. 确认无敏感信息后提交。
3. 提交信息使用 Conventional Commits：
  - `feat(module): ...`
  - `fix(module): ...`
  - `refactor(module): ...`
  - `test(module): ...`
  - `docs(module): ...`

---

## 4. 可复现标准（项目级）

项目必须长期满足以下复现标准：

1. 依赖服务可一键启动：`docker compose up -d`。
2. 前后端可按文档命令独立启动。
3. 健康检查接口可用，关键 API 可调用。
4. 测试命令稳定通过（后端测试 + 前端构建）。
5. 新人可在空环境按 README/文档在 30 分钟内跑通核心链路。

## 6.7 M6 AI 助手与 RAG 模块

### 模块目标

实现可流式输出、可追溯来源、可持久会话的 AI 助手闭环。

### 功能计划

- M6-1 SSE 对话接口
  - `delta/done/error` 事件协议与前端增量渲染。
- M6-2 会话管理
  - sessionId 生成、消息持久化（Redis + MySQL 策略可分层推进）。
- M6-3 检索增强回答
  - 结合站内检索结果生成回答上下文。
- M6-4 引用回传
  - 返回 `sourceType/sourceId/title/url`。
- M6-5 稳定性控制
  - 超时、限流、错误兜底、可观测日志。

### 设计基线（AskUserQuestion 已确认）

- 架构：`ChatClient + Agent Graph`（本期先交付 ChatClient 主链路，Graph 预留 Checkpointer 扩展位）。
- 记忆：`MessageWindowChatMemory + Summary`，窗口建议 20~30 条。
- 持久化：`Redis + JDBC(MySQL)` 热冷分层。
- 隔离：`conversationId = userId + ":" + sessionId`（Graph 对应 `threadId`）。
- 长期记忆：`RAG + 用户画像`。
- 安全：输入校验 + 限流 + 失败降级（重试后可解释报错）。

### 测试计划

- SSE 可持续返回并正常结束。
- 引用字段结构完整，可跳转。
- 异常场景返回 `error` 事件，不静默失败。

### 提交门禁

- 提交：`feat(m6): implement ai streaming and reference-backed answers`。

# MyWeb 开发计划文档（分模块迭代版）

**版本**: v1.1  
**日期**: 2026-03-24  
**依据**: `需求文档.md`、`tech-spec.md`

---

## 1. 目标与执行原则

本计划遵循“**先搭整体框架，再按模块逐个实现**”的开发方式，避免一次性并行开发所有功能导致联调失控。

- 先完成前后端基础框架与联调链路，确保系统可启动、可访问、可部署。
- 按模块顺序迭代开发，每次只推进一个主模块（可含少量配套子功能）。
- 每个模块完成后必须执行测试，测试通过后再进行 Git 提交。
- 模块未通过测试不得进入下一个模块开发。

---

## 2. 模块定义（按业务与技术分层）

## 2.1 基础框架模块（M0）

- 前端工程骨架：Vue3 + Vite + Router + Pinia + Axios + SCSS 变量体系
- 后端工程骨架：Spring Boot 3.5.12 + Java 21 + 分层目录结构（controller/service/repository 等）
- 基础中间件连接：MySQL / Redis / Elasticsearch / MinIO（先连通，后深化）
- 基础部署与反向代理：Docker Compose + Nginx
- 基础规范：统一响应体、错误码、环境变量、日志、CORS、安全基础配置

## 2.2 内容展示模块（M1）

- 首页（`/`）：简介、精选项目、最新博客入口、AI 入口占位
- 关于我（`/about`）：个人介绍、技能标签、统计信息
- 经历时间线（`/experience`）：时间轴数据展示
- 联系方式（`/contact`）：静态联系方式 + 表单框架

## 2.3 项目管理与展示模块（M2）

- 前台：项目列表、筛选、详情展示、外链跳转
- 后台：项目 CRUD（含封面、技术栈、简介、链接）
- 文件引用：项目封面与附件通过对象存储 URL 引用

## 2.4 博客管理与展示模块（M3）

- 前台：博客列表、分页、分类/标签筛选、详情页（Markdown 渲染）
- 后台：博客 CRUD、草稿/发布、基础编辑能力
- 阅读信息：发布日期、标签等元信息展示

## 2.5 搜索模块（M4）

- Elasticsearch 全文索引建立
- 博客/项目写入双写（MySQL 权威 + ES 最终一致）
- 搜索接口（关键词检索 + 高亮 + 结果分页）
- Outbox/重试/告警与索引重建入口（至少保留管理入口或脚本）

## 2.6 文件管理模块（M5）

- 文件上传（图片/Markdown/文本）
- MinIO 存储与访问 URL 生成
- 文件列表、删除、业务引用关系管理
- 严格遵守 S3 兼容契约，避免绑定 MinIO 私有语义

## 2.7 AI 助手与 RAG 模块（M6）

- DashScope 对话能力接入（Spring AI Alibaba）
- SSE 流式对话接口（`delta/done/error`）
- 向量索引构建（Embedding 固定 768 维，独立向量索引）
- 混合检索（向量 + BM25）与 RRF 融合
- 引用回溯（`sourceType/sourceId/title/url`）与会话存储（Redis + MySQL）

## 2.8 动效与体验优化模块（M7）

- 路由过渡、滚动动画、卡片悬浮等轻量动画
- 响应式和移动端适配
- 性能优化（懒加载、资源压缩、接口性能基线）

## 2.9 生产部署与运维模块（M8）

- 生产环境部署脚本与配置
- HTTPS、鉴权、限流、上传校验、安全基线
- 监控告警、日志留存、资源评估（2C4G 风险与 ES 拆分策略）

## 3. 开发顺序（严格执行）

开发顺序采用“主干优先 + 单模块闭环”：

1. `M0` 基础框架模块（必须先完成）
2. `M1` 内容展示模块
3. `M2` 项目管理与展示模块
4. `M3` 博客管理与展示模块
5. `M4` 搜索模块
6. `M5` 文件管理模块
7. `M6` AI 助手与 RAG 模块
8. `M7` 动效与体验优化模块

## 4. 每个模块的标准执行流程（DoD）

每个模块都必须按以下流程推进，不可跳步；统一采用 `任务拆分 -> 实现 -> 自测 -> 联调 -> 回归 -> 提交`：

1. **任务拆分**：将模块拆成 2~5 个可落地功能点（接口、页面、数据、联调）。
2. **实现**：仅实现当前模块范围，不引入下个模块逻辑。
3. **本地自测**：
  - 前端：页面可访问、主要交互可用、无阻断报错。
  - 后端：接口可调用、参数校验有效、异常响应符合统一格式。
  - 数据层：MySQL/ES/Redis/MinIO 相关操作可执行且结果正确。
4. **联调验证**：前后端联调通过，关键路径走通（列表、详情、提交、查询等）。
5. **回归检查**：不影响已完成模块（至少冒烟验证首页/导航/核心接口）。
6. **Git 提交**：仅在测试通过后提交，提交信息需体现模块与变更目的。

## 5. 测试与提交流程（强制）

## 5.1 模块完成后的最低测试清单

- 启动验证：前端可启动、后端可启动、依赖服务可用。
- 功能验证：本模块核心功能逐条通过。
- 接口验证：成功、失败、边界参数至少各 1 组。
- 日志验证：无持续报错、无明显异常堆栈。
- 数据验证：写入/更新/删除后数据与索引一致性符合预期（允许最终一致的模块需确认重试路径）。

## 5.2 Git 提交规则

- 原则：**先测后提，未通过不提交**。
- 建议粒度：一个提交对应一个可验证功能点或一个小型子模块。
- 提交前检查：
  - 代码可运行
  - 测试通过
  - 无敏感信息（密钥、密码）入库
- 提交信息建议模板：
  - `feat(module): add xxx capability`
  - `fix(module): resolve xxx issue`
  - `refactor(module): simplify xxx logic`

## 7. 风险与控制

- **风险 1（并行过多）**：同时做多个模块导致问题定位困难。  
**控制**：严格“一次一个主模块”，最多并行处理同模块内前后端配套。
- **风险 2（测试缺失）**：功能看似完成但不可运行。  
**控制**：执行第 5 节测试清单，不通过不进入下一模块。
- **风险 3（2C4G 资源紧张）**：ES + JVM 占用过高。  
**控制**：开发环境保守参数，生产优先升配或 ES 独立部署。
- **风险 4（AI 成本与稳定性）**：Embedding 与对话调用费用不可控。  
**控制**：限制 topK、控制切块数量、增加缓存与限流。
