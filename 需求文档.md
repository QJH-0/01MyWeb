# MyWeb 个人网站需求文档

---

## 一、项目概述

### 1.1 项目背景
构建一个综合型个人网站，用于展示个人介绍、技术博客、项目作品及职业经历，打造个人品牌。

### 1.2 网站定位
**综合型个人主页** — 集个人介绍、博客文章、项目展示于一体的在线名片

### 1.3 目标用户
- 潜在雇主/招聘方
- 技术同行/开发者社区
- 对博客内容感兴趣的读者

## 四、功能需求

### 4.1 核心功能

#### 4.1.1 首页
- 顶部导航栏（固定定位，滚动时背景变化）
- 个人简介区块（姓名、头衔、一句话介绍）
- 精选项目展示（3～4 个重点项目）
- 最新博客文章（2～3 篇）
- AI 智能助手入口（悬浮聊天按钮）
- 底部版权信息

#### 4.1.2 关于我
- 个人照片展示
- 详细个人简介
- 技能标签云或列表
- 个人数据统计（如工作年限、项目数等）
- 可选：兴趣爱好展示

#### 4.1.3 项目作品
- 项目卡片网格布局（响应式）
- 每个项目包含：封面图、标题、简介、技术栈标签
- 项目分类筛选
- 项目详情（弹窗或独立页）
- 外部链接（GitHub、在线演示等）

#### 4.1.4 博客文章（前台）
- 文章列表与分页
- 按分类、标签筛选
- Elasticsearch 全文搜索
- 文章详情（Markdown 渲染）
- 元信息：发布日期、阅读时间、标签
- 搜索结果关键词高亮

#### 4.1.5 博客与管理后台
- 文章发布、编辑、删除
- 文章双写：MySQL 权威数据 + Elasticsearch 搜索索引
- 管理后台认证：JWT + Spring Security（管理端接口启用鉴权与限流）
- 富文本或 Markdown 编辑器
- 草稿与自动保存（按实现择一或组合）

#### 4.1.6 经历时间线
- 垂直时间轴布局
- 每节点：时间段、职位/事件、描述
- 类型区分：工作、教育、项目等

#### 4.1.7 AI 智能助手
- 悬浮聊天窗口（SSE 流式回复，增量渲染）
- **RAG**：向量语义检索（Spring AI `ElasticsearchVectorStore` + DashScope `text-embedding-v3`，向量维度固定 768）为主；混合检索使用 BM25 旁路并用 **RRF** 融合（vector 默认 `topK=6` + `similarityThreshold=0.75`，BM25 默认 `topK<=4`），再按 `sourceId` 去重与截断总 token，最终写入提示词由通义千问生成回答（详见 `tech-spec.md` 第 3.4～3.5 节）
- **知识库（Knowledge Base）**：覆盖网站所有对外公开内容（前台页面与公开条目），确保仅纳入公开内容；支持发布/更新触发索引更新，删除触发清理，允许最终一致。
- **切块与元数据**：Markdown/纯文本解析并切块入库；图片默认不做 OCR，不直接 embedding（向量库仅保留图片对应可公开 URL 作为来源引用）；对每类内容按页面类型组织成可嵌入的“摘要/关键字段块”；默认单块目标约 500 tokens，overlap 10%-20%，每个 `sourceId` 最大块数建议 <= 12。写入 `sourceType/sourceId/title/url/chunkIndex` 等元数据；向量索引只存“摘要/关键字段 + 元数据”，不存全文级别的详细内容。  
- **检索返回的语义**：向量检索主要用于定位候选来源与摘要线索；当需要更准确回答时，可通过 `sourceId` 再从全文索引（BM25）或业务库拉取详情（作为可选步骤，不强依赖向量里存全文）。
- **向量检索参数**：默认 `topK=6`，默认 `similarityThreshold=0.75`（cosine 语义下过滤弱相关）；并支持 metadata 过滤（如按 `sourceType` 白名单限定公开源集合）。
- **重建策略**：当切块策略或 embedding 模型/维度变更时，支持全量重建向量索引（并可并行进行以降低对外影响）。
- 推荐相关文章与项目
- 对话历史（见技术规范中的 Redis + MySQL 方案）
- 快捷问题入口

#### 4.1.8 文件管理
- 上传：图片、Markdown 等常用类型
- 文件存储：统一 S3 兼容契约（MinIO 先落地、后续可无缝切换到 rustfs），并由后端生成访问/下载 URL、固定 Key 命名与 RFC3986 编码规则，避免把 MinIO 私有语义“写进业务逻辑”导致迁移成本不可控
- 知识库入库：Markdown/文本可切块入库；图片默认不做 OCR（仅作为来源引用）
- 文件列表与管理
- 在文章、项目中引用已上传文件

#### 4.1.9 联系方式
- 联系表单（姓名、邮箱、留言）
- 防刷：honeypot + IP/邮箱 rate limit，必要时启用 Turnstile/验证码（可配置）
- 邮箱展示
- 社交媒体链接（GitHub、LinkedIn 等）

### 4.2 交互需求
- 平滑滚动导航
- 页面切换过渡动画
- 卡片悬停效果（轻微上浮与阴影）
- 滚动触发淡入动画（GSAP）
- AI 助手回复打字机或流式展示（SSE 流式，增量渲染）
- 搜索交互与加载状态提示

### 4.3 技术实践目标（与实现阶段对应）
- Elasticsearch：索引设计、搜索与高亮、与业务双写及补偿策略
- Spring AI Alibaba：对话、DashScope 嵌入、与 Spring AI `ElasticsearchVectorStore` 的 RAG 检索增强
- Docker Compose：前端静态资源、后端、MySQL、Redis、ES、MinIO、Nginx 联调
- MinIO：上传、权限、持久化卷与备份注意点

## 五、视觉设计规范

### 5.1 设计风格
- **整体风格**：创意/艺术风格
- **设计理念**：简洁不简单，有个性但不浮夸

### 5.2 配色方案
采用**淡雅柔和**的配色（参考方向）：

| 用途 | 建议色系 |
|------|----------|
| 主色调 | 莫兰迪色系（灰蓝、灰粉、灰绿等低饱和度颜色） |
| 背景色 | 米白/浅灰白、极淡的暖色 |
| 文字色 | 深灰（非纯黑，如 #333、#2c3e50） |
| 强调色 | 一个柔和的亮色用于按钮和链接 |

实现时在全局 CSS 变量中落地主色、辅助色、背景、文字、边框与功能色，具体色值见 `tech-spec.md` 配色表。

### 5.3 动画规范
- **原则**：适度、克制，不干扰阅读
- **建议**：页面元素淡入、卡片悬停轻微上浮（约 4px）、按钮过渡、平滑滚动
- **避免**：复杂路径动画、频繁闪烁、过长加载动画

## 七、开发计划

### 7.1 第一阶段：基础框架 + 基础设施（2～3 周）
- 搭建 Vue 3 + Vite 项目（SCSS 变量、莫兰迪配色）
- 搭建 **Spring Boot 3.4.x** + Java 21 项目
- 配置 MySQL、Redis、MinIO 的 Docker 环境
- 路由与基础布局（响应式导航）
- 首页与关于我页面及对应 API
- 编写 docker-compose 供本地联调

### 7.2 第二阶段：核心功能 + Elasticsearch（约 2 周）
- 项目展示 CRUD 与前台列表
- 博客管理后台与前台列表、详情
- Elasticsearch 集成（索引、双写、失败补偿或重试）
- 全文搜索与高亮
- 经历时间线、联系方式与表单
- MinIO 文件上传与引用

### 7.3 第三阶段：AI 智能体（1～2 周）
- 集成 **Spring AI Alibaba 1.1.2.2** 与 DashScope（通义千问）
- RAG：向量索引异步构建（DashScope `text-embedding-v3` + 768 维）+ 对话侧混合检索（向量 + BM25）并用 **RRF** 融合（vector 默认 `topK=6` + `similarityThreshold=0.75`，BM25 默认 `topK<=4`）+ ChatClient 对话
- 前端聊天组件（悬浮窗、会话标识、SSE 增量渲染）
- Chat SSE API（Spring MVC + `SseEmitter` 返回 `text/event-stream`，按 `delta/done/error` 事件流式输出）、推荐相关条目、历史持久化策略

### 7.4 第四阶段：动效与生产部署（约 1 周）
- GSAP、路由过渡、滚动淡入
- 移动端适配与性能（分包、懒加载）
- 生产部署；HTTPS；日志与基本监控
- **资源**：单机 2 核 4G 时建议开发/轻量生产；ES 与 JVM 堆叠易吃满内存，生产优先考虑升配或 ES 独立部署

### 7.5 第五阶段：后端扩展（可选）
- 评论、访问统计、管理端权限
- SEO（SSR 或预渲染）
