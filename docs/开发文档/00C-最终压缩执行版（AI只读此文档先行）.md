# MyWeb 最终压缩执行版（AI 先读）

> 目的：把所有开发约束压缩为单页执行规则，避免模型自由发挥。  
> 规则强度：最高。与本文冲突时，其他文档必须回写修正。  
> 生效范围：M0~M8 全阶段。

---

## 1. 执行入口（固定）

每次开发回合必须按以下顺序读取并执行（强调：按需查、少读文档）：

1. `00E-执行总控看板（唯一入口）.md`（确认当前阶段与本轮目标）
2. 本文档（`00C`）
3. 当前阶段文档（`01`~`08`，如 `01-M0`）
4. 仅当遇到不确定/冲突时，按需引用：`需求文档.md`、`API文档与系统架构.md`、`tech-spec.md`
5. `99-阶段验收清单.md`（用于验收与证据要求）

---

## 2. 唯一权威口径（固定）

- **需求范围与边界**：`需求文档.md`
- **阶段顺序与必做动作**：`00C`（本文件）
- **接口契约（URL/方法/参数/响应/鉴权/分页）**：`API文档与系统架构.md`
- **技术选型与非功能规范**：`tech-spec.md`
- **阶段任务与验收打勾**：`01`~`08` + `99`

冲突处理（固定）：优先级从高到低为 `00C` > `API文档与系统架构.md` > 阶段文档（`01`~`08`）/ `99`（验收口径）> `tech-spec.md` > `需求文档.md`。禁止“按当前代码现状自行解释”。

---

## 3. 固定开发顺序（禁止跳阶段）

`M0 -> M1 -> M2 -> M3 -> M4 -> M5 -> M6 -> M7 -> M8`

每阶段固定流程：

`任务拆分 -> 实现 -> 自测 -> 联调 -> 回归 -> 文档更新 -> 阶段提交`

提交要求补充（强制）：

- 每完成一个“可独立验收且测试通过”的功能小任务，必须在当前回合完成一次 Git 提交（Conventional Commit）。
- 若用户明确要求暂不提交，必须在回合结论中记录“未提交原因 + 待提交文件范围 + 下一次提交触发条件”。
- 当用户指令为“按进度开始开发/继续开发”时，默认语义包含“本回合做到阶段提交”，禁止仅停留在“实现完成但未提交”状态。

---

## 4. 接口契约硬约束（不可改写）

- 基础路径：`/api`
- 响应包装：`ApiResponse<T>`
- 鉴权方案：`Spring Security + JWT`（`Authorization: Bearer <accessToken>`）
- 权限边界：`/api/admin/**` 需 `ROLE_ADMIN` 且携带 `X-Admin-Token`；`/api/ai/**` 与评论写接口需登录
- AI 流式：`GET /api/ai/chat/stream`（SSE 仅 `delta/done/error`）
- 分页默认：`page=0`、`limit=10`

说明：阶段文档中不得再次定义不同默认值或不同接口方法；同时允许补充端点级细化（校验触发条件、错误场景与示例、DB/事务映射规则等）。若发现与 `API文档与系统架构.md` 的“确定项”存在冲突，以 `API文档与系统架构.md` 为准；需要对齐时，先修改阶段文档中的冲突条目（仅保留可由阶段文档补充的细化部分），再进入代码实现。禁止让阶段文档的临时设定覆盖接口契约。

---

## 5. 阶段 DoD（统一）

每阶段必须同时满足：

1. API 可用（契约一致）
2. UI 可用（主路径可达，且具备后端不可用降级）
3. 数据闭环（写入/读取/边界）
4. 自动化通过（后端 `mvn test`，前端 `npm run build`）
5. 文档更新完成（阶段文档 + `99`）

任一不满足，禁止进入下一阶段。

### 5.1 UI 降级硬约束（全阶段通用）

- 后端 API **连接失败/超时/5xx** 时，前端必须：
  - 使用**默认数据（mock/seed）**渲染页面核心区域，保证页面不白屏、主路径仍可演示
  - 在页面显著位置给出**“后端连接失败”提示**（如 Banner/Alert/Toast 均可，但必须可见且可关闭/不阻断主流程）
- 降级只覆盖“展示与演示可用”，不允许在后端不可用时伪造“写入成功”等误导性状态（写操作需明确失败提示）。

---

## 6. 明确禁止项（统一）

- 禁止新增未声明业务能力
- 禁止跨阶段混做
- 禁止绕过统一异常与响应框架
- 禁止修改既定接口契约而不先改文档
- 禁止跳过测试/回归直接提交
- 禁止硬编码密钥

---

## 7. 变更流程（固定）

若需改需求、接口、技术方案：

1. 先改文档（`需求文档.md/00C/API/tech-spec/阶段文档`）
2. 再改代码
3. 执行测试与回归
4. 更新 `99` 验收结论

未完成步骤 1，不得进入代码实现。

---
## 7A. 阶段计划细化执行流程（固定）

当需要把“阶段文档中的本阶段大任务”进一步细化为可落地的小任务/验收点时，必须按以下顺序同步与补全（禁止跳步）：

1. 先同步到“原阶段文档”（`01~08` 对应阶段文档）：把本轮新增/修改的小任务，分别写清楚**前端计划**与**后端计划**（不能只有一个端）。
2. 再同步到 `99-阶段验收清单.md`：新增/更新对应阶段的功能验收与 **E2E 验证（Playwright MCP）** 条目，确保“前后端双端交付”和“后端不可用前端降级提示”可验收。
3. 再同步到“其他阶段文档（仅限依赖相关）”：若本轮计划会影响后续阶段的入口/接口/页面占位/数据源，必须回填到对应后续阶段文档的相关条目中，避免后续阶段出现接口或页面口径缺失。
4. 最后再进入实现：严格按小任务闭环执行（实现 -> 自测 -> 联调 -> 回归 -> 文档/99 勾选 -> Git 提交）。

补充要求（硬约束）：

- 小任务描述必须包含至少两类信息：`涉及前端页面/交互` 与 `涉及后端接口/数据处理`；任一缺失都视为“不满足计划细化口径”，禁止进入代码实现。

---

## 8. AI 执行指令（可直接引用）

- 只在已声明范围内开发，不做“顺手优化”或“猜测需求”。
- 编码前先锁定：当前阶段 + 当前任务范围 + 关键类职责边界（以阶段文档/接口契约/技术规范为准）。
- 不确定/冲突时按优先级处理，禁止“按现有代码自行解释”。
- 输出结果必须能映射到当前阶段清单与验收项（并能给出最小可验证证据）。

### 8.1 任务拆分原则（避免上下文丢失）

- 每个阶段必须拆分为 **3~5 个独立小任务**，每个任务独立验收。
- 示例（M0）：
  - 任务 1：配置分层 + 中间件连通
  - 任务 2：统一响应 + 异常处理 + traceId
  - 任务 3：JWT 认证链路（register/login/refresh/logout/me）
  - 任务 4：健康检查接口 + 前端联调
  - 任务 5：测试补齐 + 文档更新
- 每完成一个任务，立即更新 `99-阶段验收清单.md` 对应项。

### 8.2 编码约束

- 与 8 节一致：本小节不再重复列举（避免上下文膨胀）。

### 8.3 上下文管理（避免遗忘）

- 每次开始新任务前，先复述：
  - 当前阶段目标（一句话）
  - 当前任务范围（3~5 个要点）
  - 禁止事项（不跨阶段、不改契约、不跳过测试）
- 每完成一个文件/接口，立即说明"已完成 X，下一步 Y"。
