# 文档优化建议与使用指南

> 基于当前文档体系的分析，提供给开发者和 AI 的使用建议

---

## 一、当前文档体系的优点

1. **执行路径清晰**：00E（入口）→ 00C（铁律）→ 当前阶段文档
2. **接口契约统一**：API 文档作为唯一契约来源，避免冲突
3. **版本锁定策略**：技术栈版本固定，避免追新风险
4. **阶段闸门机制**：每个阶段必须通过验收才能进入下一阶段

---

## 二、AI 执行时的关键问题与解决方案

### 问题 1：文档量大，AI 容易迷失或超 token

**现状**：10+ 个文档，AI 每次都要读一遍

**解决方案**：
- 让 AI 按固定顺序读取：00E（当前阶段）→ 00C（铁律）→ 当前阶段文档
- 其他文档（需求文档.md、API、tech-spec）仅在遇到冲突或不确定时引用
- 不要让 AI 一次性读取所有文档

**执行模板**：
```
第一步：读取 00E，确认当前阶段是 M0
第二步：读取 00C，理解执行铁律
第三步：读取 01-M0，理解本阶段任务
第四步：开始执行（遇到不确定时再查 API 文档）
```

### 问题 2："禁止项"容易被 AI 忽略

**现状**：文档里有很多"禁止跨阶段""禁止改契约"，但 AI 编码时容易忘记

**解决方案**：
- 每次任务开始时，让 AI 先复述：
  - 当前阶段目标（一句话）
  - 当前任务范围（3~5 个要点）
  - 禁止事项（不跨阶段、不改契约、不跳过测试）
- 已在 00C 第 8.3 节增加"上下文管理"要求

**执行模板**：
```
开始 M0-任务1 前，AI 先说：
- 目标：配置分层 + 中间件连通
- 范围：application.yml、Docker Compose、健康检查
- 禁止：不跨阶段、不改 API 契约、不跳过测试
```

### 问题 3：AI 一次性做太多，容易遗忘前面的代码

**现状**：如果让 AI 一次性开发整个 M0，它可能会忘记前面写的代码

**解决方案**：
- 每个阶段拆分为 3~5 个独立小任务
- 每个任务独立验收，完成后立即更新 99 验收清单
- 已在 00C 第 8.1 节增加"任务拆分原则"

**M0 拆分示例**：
```
任务 1：配置分层 + 中间件连通（30 分钟）
任务 2：统一响应 + 异常处理 + traceId（30 分钟）
任务 3：JWT 认证链路（1 小时）
任务 4：健康检查接口 + 前端联调（30 分钟）
任务 5：测试补齐 + 文档更新（30 分钟）
```

### 问题 4：测试标准不够具体

**现状**："自动化通过"是指单元测试？集成测试？覆盖率要求？

**解决方案**：
- 在 99 验收清单里明确每个阶段的测试清单
- 要求提供测试证据（测试类名、测试结果截图）
- 已在 99 的 M0/M1/M2/M3/M4/M6 增加"测试验收"和"手工验证"小节

**M0 测试清单示例**：
```
- [ ] 健康检查集成测试通过（测试类：HealthControllerTest）
- [ ] 认证接口集成测试通过（测试类：AuthControllerTest）
- [ ] 统一异常处理测试通过（测试类：GlobalExceptionHandlerTest）
```

## 三、推荐的 AI 执行流程

### 阶段开始前

1. AI 读取 00E，确认当前阶段
2. AI 读取 00C，理解执行铁律
3. AI 读取当前阶段文档（如 01-M0）
4. AI 复述：阶段目标、任务范围、禁止事项

### 任务执行中

1. 拆分为 3~5 个小任务
2. 每个任务：
   - 开始前：复述任务目标和范围
   - 执行中：每完成一个文件，说明"已完成 X，下一步 Y"
   - 完成后：更新 99 验收清单对应项
3. 遇到不确定时，查阅 API 文档或 tech-spec

### 阶段结束后

1. 运行测试：`mvn test` 和 `npm run build`
2. 手工验证：按 99 清单逐项验证
3. 更新文档：阶段文档 + 99 验收清单 + 00E
4. Git 提交：Conventional Commit 格式

## 四、文档冲突处理规则

### 优先级（从高到低）

1. **00C**（执行铁律）
2. **API 文档**（接口契约）
3. **阶段文档**（当前阶段细节）
4. **tech-spec**（技术选型）
5. **需求文档.md**（需求基线）

### 冲突处理流程

1. 发现冲突时，按优先级确定"以哪个为准"
2. 修改低优先级文档，消除冲突
3. 在 Git 提交信息中说明"修复文档冲突"

---

## 五、给 AI 的快速检查清单

每次开始任务前，AI 应该检查：

- [ ] 我知道当前阶段是什么（M0~M8）
- [ ] 我知道当前任务的 3~5 个要点
- [ ] 我知道哪些事情是禁止的
- [ ] 我知道要实现哪些类和接口
- [ ] 我知道测试标准是什么
- [ ] 我知道完成后要更新哪些文档

每完成一个文件后，AI 应该说明：

- [ ] 已完成：XXX 文件/接口
- [ ] 下一步：YYY 文件/接口
- [ ] 进度：当前任务 X/5

## 六、常见陷阱与避免方法

### 陷阱 1：AI 自由发挥，新增未声明功能

**避免方法**：
- 在 00C 明确"只在已声明范围内开发"
- 每次编码前，让 AI 在文档中找到对应的类名和职责
- 如果文档没有，先补文档再编码

### 陷阱 2：AI 跨阶段混做

**避免方法**：
- 在 00C 明确"禁止跨阶段混做"
- 每次任务开始前，让 AI 复述当前阶段
- 如果 AI 提到"顺便做 XX"，立即制止

### 陷阱 3：AI 改了接口契约但没更新文档

**避免方法**：
- 在 00C 明确"禁止修改既定接口契约而不先改文档"
- 如果需要改接口，先改 API 文档，再改代码
- 在 99 验收清单增加"文档验收闸门"

### 陷阱 4：AI 跳过测试直接提交

**避免方法**：
- 在 00C 明确"禁止跳过测试/回归直接提交"
- 在 99 验收清单增加"测试验收闸门"
- 要求 AI 提供测试证据（测试类名、测试结果）

## 七、文档维护建议

### 何时更新文档

1. **需求变更**：更新 需求文档.md
2. **接口变更**：更新 API 文档
3. **技术选型变更**：更新 tech-spec
4. **执行流程变更**：更新 00C
5. **类设计变更**：更新当前阶段文档（必要时同步 API/tech-spec）
6. **验收标准变更**：更新 99

### 文档更新流程

1. 先改文档
2. 再改代码
3. 执行测试与回归
4. 更新 99 验收结论

## 十、总结

你的文档体系已经很完善，主要优化方向是：

1. **减少 AI 一次性读取的文档量**（按需引用）
2. **增加任务拆分和上下文管理**（避免 AI 迷失）
3. **细化测试标准和验收证据**（提高质量）
4. **避免过早设计**（类职责以阶段文档/接口契约为准）

现在你可以开始让 AI 按照这套文档开发了。建议从 M0 开始，每个小任务独立验收。
