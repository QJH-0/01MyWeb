# M2 项目管理与展示模块（细化版）

> 约束：本文件仅定义 M2 执行清单；接口字段、响应结构、鉴权与分页口径统一以 `API文档与系统架构.md` 为准。

## 1. 模块目标

实现项目数据从后台维护到前台展示的全流程闭环，保证数据正确、状态一致、用户可访问。

## 1A. 阶段开发计划（前后端）

### 前端开发计划（M2）

- 前台：`/projects` 列表（分类筛选 + 分页 + 三态），`/projects/:id` 详情（标签/外链按钮/空态与错误态）。
- 后台：项目管理入口与页面（列表 + 新建/编辑表单 + 删除/下线操作），表单校验与错误提示清晰。
- **后端不可用降级**：前台列表/详情在后端失败时使用默认项目数据渲染并提示“后端连接失败”；后台写操作在后端失败时明确失败，不得误报成功。

### 后端开发计划（M2）

- 数据与索引：落地 `project`、`project_tag`（含 `visible`、`deleted_at`、排序字段与必要索引）。
- 管理端 CRUD：`/api/admin/projects*`（鉴权 `ROLE_ADMIN` + `X-Admin-Token`），实现列表/详情/创建/更新/删除（软删）。
- 前台查询：`/api/projects`、`/api/projects/{id}` 强制过滤 `visible=true AND deleted_at IS NULL`。
- 校验与错误：DTO Bean Validation；404/401/403/400/500 按统一错误响应返回。
- 自动化：补齐集成/单元测试覆盖创建/更新/删除/可见性/分页筛选与参数非法分支（映射到 `99` 的 M2 条目）。

### 联调与验收里程碑（M2）

- 里程碑 1：后端管理端 CRUD 可用且权限边界正确。
- 里程碑 2：前台列表/详情与后台数据一致（可见性、软删规则正确）。
- 里程碑 3：构建与测试证据回填到 `99-阶段验收清单.md` 的 `M2` 区域，完成阶段闸门 A~E。

## 2. 功能清单

### 2.1 数据层

- `project` 主表：标题、摘要、分类、标签、封面、链接、可见性、时间戳。
- `project_tag` 子表（或等效结构）：支持多标签。

### 2.2 后端接口层

- 管理端：
  - `GET /api/admin/projects`（分页、分类筛选）
  - `GET /api/admin/projects/{id}`
  - `POST /api/admin/projects`
  - `PUT /api/admin/projects/{id}`
  - `DELETE /api/admin/projects/{id}`（软删除或下线）
- 前台：
  - `GET /api/projects`
  - `GET /api/projects/{id}`

### 2.3 前端展示层

- 前台项目列表：分类筛选、空状态、加载状态。
- 前台项目详情：核心信息、标签、外链按钮。
- 后台管理页：项目创建/编辑/删除、校验提示。

### 2.4 业务规则

- 不可见项目不得在前台查询到。
- 无效参数返回统一错误结构。
- 删除后的项目前台不可访问。

## 3. 数据库表结构

### 3.1 Project 实体

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| id | BIGINT | 是 | 主键，自增 |
| title | VARCHAR(200) | 是 | 项目标题 |
| summary | TEXT | 否 | 项目摘要 |
| description | TEXT | 否 | 项目描述（Markdown） |
| category | VARCHAR(50) | 否 | 分类 |
| cover_url | VARCHAR(500) | 否 | 封面图 URL |
| project_url | VARCHAR(500) | 否 | 项目链接 |
| source_url | VARCHAR(500) | 否 | 源码链接 |
| visible | TINYINT(1) | 否 | 是否可见：0-否 1-是，默认 1 |
| sort_order | INT | 否 | 排序权重，默认 0 |
| created_at | DATETIME | 否 | 创建时间 |
| updated_at | DATETIME | 否 | 更新时间 |
| deleted_at | DATETIME | 否 | 删除时间（软删除） |

索引：category、visible、sort_order

### 3.2 ProjectTag 实体

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| id | BIGINT | 是 | 主键，自增 |
| project_id | BIGINT | 是 | 项目 ID |
| tag | VARCHAR(50) | 是 | 标签 |

索引：project_id、tag

## 4. 后端接口与 DTO 契约

### 4.1 前台接口

#### 4.1.1 项目列表

- 接口路径：`GET /api/projects`
- 请求参数：category（分类筛选）、page、limit（默认值以 `API文档与系统架构.md` 为准）
- 返回数据：list（项目列表）、total（总数）、page、limit

#### 4.1.2 项目详情

- 接口路径：`GET /api/projects/{id}`
- 返回数据：项目完整信息（包含 description、tags、sourceUrl 等）

### 4.2 管理端接口

#### 4.2.1 管理项目列表

- 接口路径：`GET /api/admin/projects`
- 请求参数：category、visible、page、limit

#### 4.2.2 创建项目

- 接口路径：`POST /api/admin/projects`
- 请求体字段：title（必填）、summary、description、category、coverUrl、projectUrl、sourceUrl、visible、sortOrder、tags
- 校验规则：标题必填且最长 200 字符，URL 需符合 http/https 格式，标签最多 10 个

#### 4.2.3 更新项目

- 接口路径：`PUT /api/admin/projects/{id}`
- 请求体：同创建请求，可部分更新

#### 4.2.4 删除项目

- 接口路径：`DELETE /api/admin/projects/{id}`
- 软删除：仅更新 deleted_at 字段

### 4.3 错误响应与错误码

- `ApiResponse.error` 语义与错误码表：见 `错误码与错误响应.md`
- 端点级错误触发条件与返回示例：见下节“端点级 API 细化”

### 4.4 端点级 API 细化（字段约束 / DB 映射 / 错误场景）

> 说明：本节只做“可落地细化”；确定项（URL/方法/分页默认/鉴权/响应结构外层）仍以 `API文档与系统架构.md` 为准。

#### 4.4.1 `GET /api/projects`（公开）

- 请求参数：
  - `category?`: 分类筛选（非法值返回 `VALIDATION_ERROR`）
  - `page?`/`limit?`: 默认值以 `API文档与系统架构.md` 为准
- 数据过滤口径（与前台可见性一致）：
  - `visible = true`
  - `deleted_at IS NULL`
- 响应结构：
  - `ApiResponse<PagedResult<ProjectResponseDTO>>`
- `ProjectResponseDTO` -> `project` / `project_tag` 映射：
  - `id` -> `project.id`
  - `title` -> `project.title`
  - `summary` -> `project.summary`
  - `description?` -> `project.description`
  - `category?` -> `project.category`
  - `tags` -> 从 `project_tag.tag` 聚合得到（聚合源：`project_tag.project_id = project.id`）
  - `coverUrl?` -> `project.cover_url`
  - `projectUrl?` -> `project.project_url`
  - `sourceUrl?` -> `project.source_url`
  - `visible` -> `project.visible`
  - `sortOrder?` -> `project.sort_order`
  - `createdAt` -> `project.created_at`
  - `updatedAt` -> `project.updated_at`
  - `deletedAt?` -> `project.deleted_at`
- 错误场景（示例）：
  - 参数校验失败（如类型不匹配）：`400 VALIDATION_ERROR`
  - 发生未预期错误：`500 INTERNAL_ERROR`

#### 4.4.2 `GET /api/projects/{id}`（公开详情）

- 数据过滤口径：
  - `visible = true`
  - `deleted_at IS NULL`
- 响应结构：
  - `ApiResponse<ProjectResponseDTO>`
- 错误场景（示例）：
  - id 不存在或不可见/已删除：`404 NOT_FOUND`
  - 发生未预期错误：`500 INTERNAL_ERROR`

#### 4.4.3 `GET /api/admin/projects`（后台列表）

- 鉴权：
  - 需 `Authorization: Bearer <accessToken>` 且 `ROLE_ADMIN`
  - 需携带 `X-Admin-Token`
- 请求参数：
  - `category?`：分类筛选
  - `visible?`：可见性筛选（可选）
  - `page?`/`limit?`：默认值以 `API文档与系统架构.md` 为准
- 数据过滤口径（后台）：
  - 默认仅返回 `deleted_at IS NULL`
  - 若传入 `visible`，则额外筛选 `visible = <visible>`
- 响应结构：
  - `ApiResponse<PagedResult<ProjectResponseDTO>>`
- 错误场景（示例）：
  - 缺失/无效 JWT：`401 UNAUTHORIZED`
  - 缺失 `X-Admin-Token` 或权限不足：`403 FORBIDDEN`
  - 参数校验失败：`400 VALIDATION_ERROR`

#### 4.4.4 `GET /api/admin/projects/{id}`（后台详情）

- 鉴权：同 `4.4.3`
- 数据过滤口径：
  - `deleted_at IS NULL`（已软删除/不存在返回 404）
- 响应结构：
  - `ApiResponse<ProjectResponseDTO>`
- 错误场景（示例）：
  - id 不存在或已删除：`404 NOT_FOUND`
  - 权限错误：`401/403`（见 `错误码与错误响应.md`）

#### 4.4.5 `POST /api/admin/projects`（后台创建）

- 鉴权：同 `4.4.3`
- 请求体字段（关键约束）：
  - `title` 必填，<= 200
  - `summary` 必填于底座 DTO 时以 `API文档与系统架构.md` 为准（阶段仅约束不放宽）
  - `tags` 必填，最多 10 个，单元素 <= 60（以 `API文档与系统架构.md` 为准）
  - `visible` 必填（boolean）
  - `coverUrl/projectUrl/sourceUrl?`：<= 1000
- 数据落库口径：
  - 写入 `project` 表基本字段
  - 写入 `project_tag` 表（多条，每个 tag 一行）
  - 初始化 `deleted_at = NULL`，其余时间字段由服务端维护
- 响应结构：
  - `ApiResponse<ProjectResponseDTO>`
- 错误场景（示例）：
  - 校验失败（字段缺失/长度超限/visible 非 boolean）：`400 VALIDATION_ERROR`
  - 权限错误：`401/403`
  - 同一字段唯一约束冲突（如实现后加唯一约束）：`400 VALIDATION_ERROR`

#### 4.4.6 `PUT /api/admin/projects/{id}`（后台更新）

- 鉴权：同 `4.4.5`
- 数据落库口径：
  - 更新 `project` 基本字段
  - `tags`：建议采用“先删后写”或差量更新（以实现为准，但最终以请求 tags 为准）
- 响应结构：
  - `ApiResponse<ProjectResponseDTO>`
- 错误场景：
  - id 不存在或已删除：`404 NOT_FOUND`
  - 校验失败：`400 VALIDATION_ERROR`

#### 4.4.7 `DELETE /api/admin/projects/{id}`（后台删除/下线）

- 鉴权：同 `4.4.3`
- 删除策略（与阶段 DB 约束一致）：
  - 只更新 `deleted_at`（软删除），不物理删除
- 响应结构：
  - `ApiResponse<null>`
- 错误场景（示例）：
  - id 不存在或已删除：`404 NOT_FOUND`
  - 权限错误：`401/403`

## 8A. 固定技术实现方案（禁止自由发挥）

### 8A.1 固定实现顺序

1. 先落地 `project`/`project_tag` 数据结构与索引。
2. 再实现管理端接口（列表->详情->创建->更新->删除）。
3. 再实现前台接口（列表->详情）并接入可见性过滤。
4. 前端最后联调（后台管理页 + 前台展示页）。
5. 完成集成测试与构建回归。

### 8A.2 固定技术方案

- 持久化：JPA Repository + Service 分层，禁止在 Controller 直接操作 Repository。
- 删除策略：固定软删除（`deleted_at`），禁止物理删除。
- 可见性策略：前台查询必须强制 `visible=true AND deleted_at IS NULL`。
- 参数校验：DTO 上使用 Bean Validation，错误统一返回 `BAD_REQUEST`。
- 响应结构：统一 `ApiResponse`，禁止返回裸列表。
- 内容主源：项目内容的权威来源在开发/测试默认为本地 `content/projects/*.md`；生产模式为 GitHub 对应目录。后台创建/编辑最终必须写回 Markdown（见 `tech-spec.md` 3.1A），MySQL 为派生结构化存储，禁止反向覆盖 Markdown。

### 8A.3 通用约束引用（固定）

- 本阶段“验证方式 / 禁止项 / 提交要求”统一执行 `00C-最终压缩执行版（AI只读此文档先行）.md`，不在本文件重复定义。
