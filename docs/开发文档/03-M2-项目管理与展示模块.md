# M2 项目管理与展示模块（细化版）

> 约束：本文件仅定义 M2 执行清单；接口字段、响应结构、鉴权与分页口径统一以 `API文档与系统架构.md` 为准。

## 1. 模块目标

实现项目数据从后台维护到前台展示的全流程闭环，保证数据正确、状态一致、用户可访问。

## 2. 功能清单

### 2.1 数据层

- `project` 主表：标题、摘要、分类、标签、封面、链接、可见性、时间戳。
- `project_tag` 子表（或等效结构）：支持多标签。

### 2.2 后端接口层

- 管理端：
  - `GET /api/admin/projects`（分页、分类筛选）
  - `GET /api/admin/projects/{id}`
  - `POST /api/admin/projects`
  - `PUT /api/admin/projects/{id}`
  - `DELETE /api/admin/projects/{id}`（软删除或下线）
- 前台：
  - `GET /api/projects`
  - `GET /api/projects/{id}`

### 2.3 前端展示层

- 前台项目列表：分类筛选、空状态、加载状态。
- 前台项目详情：核心信息、标签、外链按钮。
- 后台管理页：项目创建/编辑/删除、校验提示。

### 2.4 业务规则

- 不可见项目不得在前台查询到。
- 无效参数返回统一错误结构。
- 删除后的项目前台不可访问。

## 3. 数据库表结构

### 3.1 Project 实体

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| id | BIGINT | 是 | 主键，自增 |
| title | VARCHAR(200) | 是 | 项目标题 |
| summary | TEXT | 否 | 项目摘要 |
| description | TEXT | 否 | 项目描述（Markdown） |
| category | VARCHAR(50) | 否 | 分类 |
| cover_url | VARCHAR(500) | 否 | 封面图 URL |
| project_url | VARCHAR(500) | 否 | 项目链接 |
| source_url | VARCHAR(500) | 否 | 源码链接 |
| visible | TINYINT(1) | 否 | 是否可见：0-否 1-是，默认 1 |
| sort_order | INT | 否 | 排序权重，默认 0 |
| created_at | DATETIME | 否 | 创建时间 |
| updated_at | DATETIME | 否 | 更新时间 |
| deleted_at | DATETIME | 否 | 删除时间（软删除） |

索引：category、visible、sort_order

### 3.2 ProjectTag 实体

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| id | BIGINT | 是 | 主键，自增 |
| project_id | BIGINT | 是 | 项目 ID |
| tag | VARCHAR(50) | 是 | 标签 |

索引：project_id、tag

## 4. 后端接口与 DTO 契约

### 4.1 前台接口

#### 4.1.1 项目列表

- 接口路径：`GET /api/projects`
- 请求参数：category（分类筛选）、page、limit（默认值以 `API文档与系统架构.md` 为准）
- 返回数据：list（项目列表）、total（总数）、page、limit

#### 4.1.2 项目详情

- 接口路径：`GET /api/projects/{id}`
- 返回数据：项目完整信息（包含 description、tags、sourceUrl 等）

### 4.2 管理端接口

#### 4.2.1 管理项目列表

- 接口路径：`GET /api/admin/projects`
- 请求参数：category、visible、page、limit

#### 4.2.2 创建项目

- 接口路径：`POST /api/admin/projects`
- 请求体字段：title（必填）、summary、description、category、coverUrl、projectUrl、sourceUrl、visible、sortOrder、tags
- 校验规则：标题必填且最长 200 字符，URL 需符合 http/https 格式，标签最多 10 个

#### 4.2.3 更新项目

- 接口路径：`PUT /api/admin/projects/{id}`
- 请求体：同创建请求，可部分更新

#### 4.2.4 删除项目

- 接口路径：`DELETE /api/admin/projects/{id}`
- 软删除：仅更新 deleted_at 字段

### 4.3 错误响应（唯一契约引用）

- 错误返回结构与 HTTP 状态码口径：统一以 `API文档与系统架构.md` 为准，本文件不再维护“错误码表/状态码表”以避免冲突。

## 8A. 固定技术实现方案（禁止自由发挥）

### 8A.1 固定实现顺序

1. 先落地 `project`/`project_tag` 数据结构与索引。
2. 再实现管理端接口（列表->详情->创建->更新->删除）。
3. 再实现前台接口（列表->详情）并接入可见性过滤。
4. 前端最后联调（后台管理页 + 前台展示页）。
5. 完成集成测试与构建回归。

### 8A.2 固定技术方案

- 持久化：JPA Repository + Service 分层，禁止在 Controller 直接操作 Repository。
- 删除策略：固定软删除（`deleted_at`），禁止物理删除。
- 可见性策略：前台查询必须强制 `visible=true AND deleted_at IS NULL`。
- 参数校验：DTO 上使用 Bean Validation，错误统一返回 `BAD_REQUEST`。
- 响应结构：统一 `ApiResponse`，禁止返回裸列表。
- 内容主源：项目内容的权威来源在开发/测试默认为本地 `content/projects/*.md`；生产模式为 GitHub 对应目录。后台创建/编辑最终必须写回 Markdown（见 `tech-spec.md` 3.1A），MySQL 为派生结构化存储，禁止反向覆盖 Markdown。
