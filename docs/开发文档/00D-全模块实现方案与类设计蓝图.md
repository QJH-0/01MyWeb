# MyWeb 全模块实现方案与类设计蓝图（M0-M8）

> 目的：把“需求”转成“可直接编码的实现方案与类设计”，避免模型生成偏差代码。  
> 强制口径：当阶段文档只描述目标时，必须以本文档的类设计与职责边界为落地依据。

---

## 1. 统一分层与命名约束

- Controller：只做参数接收、鉴权校验、响应包装，不写业务编排。
- Service：业务编排层，负责事务边界、领域规则、失败补偿调度。
- Repository：只负责持久化访问，不承载业务判断。
- Adapter/Client：第三方系统适配（GitHub、DashScope、MinIO/S3、Elasticsearch）。
- DTO/VO：接口入参与出参；Entity：数据库模型。

后端核心包建议：

- `com.myweb.controller`
- `com.myweb.service`
- `com.myweb.repository`
- `com.myweb.entity`
- `com.myweb.dto`
- `com.myweb.common`
- `com.myweb.ai`（RAG 编排、召回、引用构建）
- `com.myweb.search`（Outbox、索引消费、重建）
- `com.myweb.storage`（S3 兼容抽象）

## 2. M0 基础框架与联调链路

### 2.1 实现方案

1. 建立统一响应与统一异常处理。
2. 建立 traceId 注入链路，所有请求可追踪。
3. 健康检查与基础路由联调。
4. JWT 认证与鉴权链路按接口契约落地（详见 `API文档与系统架构.md`）。

### 2.2 核心类职责（仅列必须存在的类，具体实现在阶段文档）

| 类名 | 职责 | 所属包 |
|------|------|--------|
| `ApiResponse<T>` | 统一响应模型 | `common` |
| `GlobalExceptionHandler` | 统一异常到错误码映射 | `common` |
| `TraceIdFilter` | 请求入口生成/透传 traceId | `common` |
| `HealthController` | 健康检查与版本信息 | `controller` |
| `AuthController` | 注册/登录/刷新/登出/当前用户接口 | `controller` |

**说明**：具体方法签名、参数校验、异常处理细节在各阶段文档中定义，避免过早设计。

## 6. M4-M5 搜索与文件管理（重点）

### 6.1 搜索实现方案

1. 内容变更写 `es_index_outbox`，与业务事务同提交。
2. `SearchOutboxConsumer` 定时消费，失败指数退避重试。
3. 搜索查询统一融合 BM25 与业务排序（时间衰减）。

### 6.2 搜索核心类职责

| 类名 | 职责 | 所属包 |
|------|------|--------|
| `SearchController` | 搜索 API 出口 | `controller` |
| `SearchService` | 搜索编排层 | `service` |
| `SearchQueryService` | ES 查询和高亮封装 | `service` |
| `SearchOutboxPublisher` | 业务侧发布索引事件 | `service` |
| `SearchOutboxConsumer` | 后台消费索引事件 | `service` |
| `SearchIndexService` | 索引写入/删除/重建 | `service` |

### 6.3 文件管理实现方案

1. 文件元数据存 MySQL，文件体走 S3 兼容存储。
2. 下载必须走预签名 URL，避免暴露存储内网地址。
3. 上传严格校验类型、大小、权限。

### 6.4 文件核心类职责

| 类名 | 职责 | 所属包 |
|------|------|--------|
| `AdminFileController` | 后台文件上传/列表/删除 | `controller` |
| `FilePublicController` | 前台文件下载 | `controller` |
| `FileStorageService` | 文件存储业务 | `service` |
| `ManagedFileRepository` | 文件元数据持久化 | `repository` |
| `ObjectStorageClient` | S3 兼容存储接口 | `storage` |
| `MinioStorageClient` | MinIO 实现 | `storage` |

**说明**：具体实现细节在 `04-M4` 和 `05-M5` 阶段文档中定义。

## 7. M6 AI 助手与 RAG（重点）

### 7.1 实现方案

1. 对话入口为 SSE：`delta/done/error`。
2. 召回固定为混合检索：BM25 + 向量，融合策略 RRF。
3. 会话分层：Redis 短期上下文 + MySQL 长期消息。
4. 回答必须返回可追溯来源（`sourceType/sourceId/title/url`）。
5. 模型超时降级到“仅检索回答”。

### 7.2 类设计（必须存在）

- `AiController`
- `AiAssistantService`
- `AiOrchestrator`
- `RetrievalService`
- `CitationBuilder`
- `ChatSessionCache`（Redis）
- `ChatSessionRepository`（MySQL）
- `ChatMessageRepository`
 
### 7.3 方法签名与参数校验

方法签名、入参/出参结构与参数校验细节必须在对应阶段文档（如 `06-M6`）落地定义；本文件仅给出“类职责与边界”，避免过早设计导致上下文膨胀。
