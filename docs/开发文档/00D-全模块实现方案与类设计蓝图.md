## 1. 统一分层与命名约束

- Controller：只做参数接收、鉴权校验、响应包装，不写业务编排。
- Service：业务编排层，负责事务边界、领域规则、失败补偿调度。
- Repository：只负责持久化访问，不承载业务判断。
- Adapter/Client：第三方系统适配（GitHub、DashScope、MinIO/S3、Elasticsearch）。
- DTO/VO：接口入参与出参；Entity：数据库模型。

后端核心包建议：

- `com.myweb.controller`
- `com.myweb.service`
- `com.myweb.repository`
- `com.myweb.entity`
- `com.myweb.dto`
- `com.myweb.common`
- `com.myweb.ai`（RAG 编排、召回、引用构建）
- `com.myweb.search`（Outbox、索引消费、重建）
- `com.myweb.storage`（S3 兼容抽象）

## 6. M4-M5 搜索与文件管理（重点）

### 6.1 搜索实现方案

1. 内容变更写 `es_index_outbox`，与业务事务同提交。
2. `SearchOutboxConsumer` 定时消费，失败指数退避重试。
3. 搜索查询统一融合 BM25 与业务排序（时间衰减）。

### 6.2 搜索类设计（必须存在）

- `SearchController`：搜索 API 出口。
- `SearchService`：搜索编排层。
- `SearchQueryService`：ES 查询和高亮封装。
- `SearchOutboxPublisher`：业务侧发布索引事件。
- `SearchOutboxConsumer`：后台消费索引事件。
- `SearchIndexService`：索引写入/删除/重建。

### 6.3 搜索关键方法签名

```java
public interface SearchService {

public interface SearchIndexService {
```

### 6.4 文件管理实现方案

1. 文件元数据存 MySQL，文件体走 S3 兼容存储。
2. 下载必须走预签名 URL，避免暴露存储内网地址。
3. 上传严格校验类型、大小、权限。

## 7. M6 AI 助手与 RAG（重点）

### 7.1 实现方案

1. 对话入口为 SSE：`delta/done/error`。
2. 召回固定为混合检索：BM25 + 向量，融合策略 RRF。
3. 会话分层：Redis 短期上下文 + MySQL 长期消息。
4. 回答必须返回可追溯来源（`sourceType/sourceId/title/url`）。
5. 模型超时降级到“仅检索回答”。

## 10. 统一错误码与异常语义

- `400` 参数校验失败：`VALIDATION_ERROR`
- `401` 登录态无效：`UNAUTHORIZED`
- `403` 权限不足：`FORBIDDEN`
- `404` 资源不存在：`NOT_FOUND`
- `429` 限流命中：`RATE_LIMITED`
- `500` 系统内部异常：`INTERNAL_ERROR`

所有异常必须由 `GlobalExceptionHandler` 归一处理，不允许直接返回堆栈信息。
