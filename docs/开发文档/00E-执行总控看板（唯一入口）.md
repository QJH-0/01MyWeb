### 4.2 文档更新与 Git 提交节奏（本仓库约定）

- **代码提交粒度（与 `00C` §3、`01common` §四 一致）**：**仅**在 **（1）一个功能开发并完成约定测试、达到可验收**，或 **（2）对某一功能点的一次较大修改且测试已通过** 时，再执行 Git 提交。**禁止**每次小改、单文件微调就提交；本地可攒改合并为一次提交。
- **先写文档、后随代码提交**：`00E` 回填、`99-阶段验收清单.md` 勾选、当前阶段文档、`开发计划.md` 等，可在功能开发与测试过程中**持续在本地修改**，用于跟踪进度与留证路径。
- **不要「改一次文档就提交一次」**：上述文档变更默认 **不与** 零散进度强绑定单独入库；应在 **本阶段/本功能点的代码实现与测试（含约定门禁）完成后**，将 **文档与代码在同一批提交**（或紧随其后、同一主题的少量提交），避免仓库里充斥仅文档的碎提交。
- **例外**：纯勘误、与当期开发无关的口径统一，可单独文档提交（建议在 message 中注明 `docs:` 与原因）。
- **与旧口径/工具提示冲突时**：以 **`01common` §四** 与 **`00C` §3** 的提交粒度为准；质量门禁（先测后提、可验收再入库）不变。
